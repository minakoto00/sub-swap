# ADR-001: AES-256-GCM for Profile Encryption

**Status:** Accepted
**Date:** 2026-04-02

## Context

sub-swap stores inactive Codex profile credentials (auth.json, config.toml) on disk. These files contain API tokens and configuration that must be protected at rest. The encryption scheme must provide both confidentiality and tamper detection — if an encrypted file is modified, decryption must fail rather than silently produce corrupted output.

## Decision

Use AES-256-GCM (Galois/Counter Mode) via the `aes-gcm` crate with:
- 256-bit key (32 bytes) generated from OS CSPRNG
- 96-bit nonce (12 bytes) randomly generated per encryption operation
- Output format: `[12-byte nonce][ciphertext][16-byte GCM authentication tag]`
- Each file encrypted independently (not bundled)
- Pure functions: `encrypt(plaintext, key) -> Result<Vec<u8>>` and `decrypt(data, key) -> Result<Vec<u8>>`

## Consequences

**Easier:**
- Tamper detection is automatic (GCM authentication tag)
- Per-operation random nonce eliminates nonce-reuse risk for reasonable usage patterns
- Pure function design makes encrypt/decrypt testable without mocking

**Harder:**
- Key loss means encrypted profiles are permanently unrecoverable
- No key rotation mechanism (would require decrypting and re-encrypting all profiles)

**Constrained:**
- Minimum ciphertext size is 28 bytes (12 nonce + 16 tag) — decrypt rejects anything smaller
- The `aes-gcm` crate is a required dependency

## Implementation

- `src/crypto/mod.rs` — `encrypt()`, `decrypt()`, `generate_key()` functions
- `src/crypto/keychain.rs` — Key storage via `KeyStore` trait
